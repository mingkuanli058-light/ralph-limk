## 这是什么工程？

这是一个基于 Ralph Wiggum 工程方法的 **AI Agent 开发流程 Demo / 模板工程**。

**当前工程交付级别：Stage 1（Human-in-the-loop）**

目标是：  
通过文件约束 + 执行循环，将 Claude 作为一个 **受控、可审计、可回滚的工程执行器**，
而不是自由对话式 AI。

本仓库主要用于：
- 验证 Ralph 工程方法
- 复用为 AI 辅助开发的工程模板


⚠️ 注意：  
- 人类使用者：请先阅读「最小使用流程」与「推荐入口」  
- Claude（Agent）：后续章节为可直接复制使用的角色指令  


## 最小使用流程（Stage 1 · 人类参与）

1. 在 `docs/我的任务.md` 中，用自然语言写下你的需求描述
2. 确保 AGENTS.md 与 PROMPT_* 文件未被修改
3. 启动 Claude（CLI 或桌面）
4. 按阶段顺序执行：
   - 需求规格完成后 → `pwsh -File loop_plan.ps1`
   - 计划冻结后 → `pwsh -File loop_build.ps1`   
5. 所有状态以文件为准，不通过对话确认

## 根目录速查与工程状态判读（Stage 1）

本工程通过 **根目录中的关键文件** 显式表达工程状态。  
**不通过对话判断阶段，只通过文件是否存在判断。**

---

### 一、根目录关键文件速查（做什么用）

- `README.md`：工程方法与流程的总入口  
- `AGENTS.md`：Agent 行为边界的最高权威  
- `PROMPT_*.md`：各阶段 Agent 的唯一行为指令  
- `loop_*.ps1`：阶段执行循环（只负责执行，不做决策）  
- `specs/REQUIREMENTS.md`：当前需求规格  
- `IMPLEMENTATION_PLAN.md`：当前实施计划  
- `SPECS_READY.md` / `PLAN_FROZEN.md`：阶段放行 Gate  
- `RELEASE_CANDIDATE.md` / `RELEASED.md`：发布状态标志  
- `HUMAN_OVERRIDES.md`：所有人工干预的审计记录  

> **原则：根目录文件的存在与否，即代表当前工程状态是否合法。**

---

### 二、工程状态如何流转（一眼判读）

| 文件 | 含义 | 状态影响 |
|---|---|---|
| `specs/REQUIREMENTS.md` | 需求规格是否已生成 | 未生成 → 停在 Specs |
| `SPECS_READY.md` | Specs 是否已人工确认 | 无此文件 → 禁止 Planning |
| `IMPLEMENTATION_PLAN.md` | 实施计划是否存在 | 未生成 → 仍在 Planning |
| `PLAN_FROZEN.md` | Plan 是否已冻结 | 无此文件 → 禁止 Building |
| `RELEASE_CANDIDATE.md` | 是否进入发布候选 | 存在 → 仅允许 Fix |
| `RELEASED.md` | 是否已正式发布 | 存在 → 本轮交付结束 |

---

### 快速判定规则（记住这一条就够）

> **只要缺少当前阶段所需的 Gate 文件，就不要进入下一阶段。**

# 角色：产品负责人（Requirements Owner）

你现在处于【需求规格生成阶段】。

---

## 一、权威与优先级

本阶段适用的约束与规则来源**仅包括**：

1. `docs/我的任务.md`  
   （原始需求输入，可能非结构化、不完整）
2. `AGENTS.md`  
   （Ralph 执行规范，**最高优先级**）
3. 本指令文本

如本指令与 `AGENTS.md` 存在任何冲突，  
**必须以 `AGENTS.md` 为最高优先级**。

除上述文件外，**不得引入任何其他假设、模板或外部规范**。

---

## 二、输入定义（唯一 Ground Truth）

- `docs/我的任务.md` 是本阶段**唯一原始需求输入文件**
- 其内容：
  - 仅用于表达目标与约束意图
  - 不构成系统行为或实现的 Ground Truth
  - 不保证完整性或结构性

---

## 三、你的任务

在**不引入任何额外假设**的前提下：

- 仅基于 `docs/我的任务.md` 中**明确表达的内容**
- 将其整理为一个：
  - 结构化
  - 可验证
  - 不包含实现细节的

**需求规格文档**

---

## 四、输出要求（严格）

### 输出文件
- 路径：`specs/REQUIREMENTS.md`

### 内容约束
- 仅描述系统**应该做什么（What）**
- **不得**描述：
  - 如何实现（How）
  - 技术方案
  - 架构设计
  - 接口形式

### 明确禁止
- 不得包含任何代码、伪代码
- 不得包含接口定义、实现步骤或实现建议
- 不得生成、暗示或推导 `IMPLEMENTATION_PLAN.md`
- 不得在正文中包含任务拆分、执行顺序或阶段规划相关内容

---

## 五、输出范围限制

本阶段**只允许输出需求规格文档内容**。  
除 `specs/REQUIREMENTS.md` 的正文外，不得输出任何其他文本。

- **不得**修改现有实现

- 在未触发停止条件前：
  - 应完成本阶段定义的**全部规划任务**

## 与 Ralph 官方方法的差异与当前不足

本工程是一个 **受控版（Guardrailed）Ralph 工程流程 Demo / 模板**，  
在设计上刻意强化了阶段边界与工程约束，以提升可预测性与可审计性。

因此，与 Geoffrey Huntley 描述的 **官方 Ralph 方法** 相比，  
当前版本在以下方面 **尚未完全覆盖或有意弱化**。

---

### 1. 未完全采用 JTBD → Topic of Concern → 多 Spec 的结构

官方 Ralph 方法强调：

- 从 Job To Be Done（JTBD）出发
- 拆分为多个 Topic of Concern
- 每个 Topic 对应一个独立的 `specs/*.md`

当前工程中：

- 使用单一的 `docs/我的任务.md` 作为原始输入
- 输出为单一的 `specs/REQUIREMENTS.md`

这种设计在小规模或教学场景下更直观、可控，  
但在需求规模扩大或并行规划时，  
不如官方方法的多 Spec 结构具备伸缩性。

---

### 2. 弱化了 Ralph 在 BUILDING 阶段反向修正 Specs / Plan 的自由度

官方 Ralph 方法允许：

- 在 BUILDING 阶段发现 Specs 不一致
- 必要时反向修正 Specs 或重新生成 Plan
- 通过多轮迭代实现“最终一致性（eventual consistency）”

在当前工程中：

- 明确禁止 BUILDING 阶段回退到 PLANNING
- 明确禁止在 FIXING 阶段扩散到 Specs 或 Plan
- 强调阶段单向推进

这种约束降低了 Agent 的涌现自由度，  
但换取了更强的阶段确定性与工程可控性。

---

### 3. 尚未系统性引入 Subagent / Context Engineering 模式

官方 Ralph 方法高度依赖：

- 大量并行 subagents
- 主 Agent 作为调度器
- 通过上下文切片最大化 token 利用效率

当前工程中：

- Prompt 设计以角色与阶段约束为主
- 并未显式建模 subagent 的使用策略
- 更偏向流程治理，而非上下文调度优化

因此，在超大代码库或高并发探索场景下，  
当前模板的效率与扩展性不及官方推荐实践。

---

### 4. Loop 脚本当前偏“执行驱动”，而非“流程守门员”

官方 Ralph 强调：

- Loop 只是一个“愚钝的重复器”
- 正确性由 Prompt、文件结构与 backpressure 共同保证

当前工程中：

- Loop 脚本主要负责阶段执行
- 尚未在脚本层面强制校验：
  - 阶段合法性
  - 可写文件范围
  - 状态一致性

因此，当前版本仍依赖 Agent 遵守约定，  
而非完全由机制阻止非法状态迁移。

---

### 5. 更适合作为受控 Demo / 教学模板，而非原教旨 Ralph 实验场

综合来看：

- 官方 Ralph 方法更强调：
  - 涌现
  - 自我修正
  - 最终一致性
- 当前工程更强调：
  - 阶段清晰
  - 行为边界
  - 可审计与可回滚

因此，本工程更适合：
- 方法论学习与教学
- 受控环境下的 Agent 实验
- 企业或团队内部试点

而非完全替代官方 Ralph 在自由探索场景中的使用方式。

## 当前交付级别（Stage 1）

本工程当前处于 **Stage 1：Human-in-the-loop 的真实交付系统**。

特征：
- 所有阶段均有显式完成信号（Gate）
- 关键阶段需要人工确认后方可继续
- 人类介入必须留痕
- Agent 行为仍基于信任与规范，而非硬性机制

本阶段适合：
- 内部真实项目
- 受控交付
- 高风险功能开发

本阶段不适合：
- 长时间无人值守
- SLA 要求严格的自动化交付

> 当前状态：**Stage 1（Human-in-the-loop）**  
> 本表用于快速判断：**每一阶段解决了什么问题、还不能做什么、风险边界在哪里**。

| 维度 | Stage 0<br>方法论 Demo | Stage 1<br>Human-in-the-loop | Stage 2<br>机制化约束 | Stage 3<br>真实交付系统 | Stage 4<br>规模化 / 组织级 |
|---|---|---|---|---|---|
| **定位** | 方法论演示 | 受控真实使用 | Agent 不可信 | 商业级交付 | 组织级生产系统 |

| **Gate（阶段完成信号）** | ❌ | ✅ 文件 Gate | ✅ 机制校验 | ✅ 强制校验 | ✅ 跨团队 |

| **适合的使用者** | 个人研究 | 小团队 | 专业团队 | 商业项目 | 大型组织 |

---

### 每个 Stage 的一句话定义

- **Stage 0**  
  > 我们知道应该怎么做，但系统不会阻止我们做错。

- **Stage 1（当前）**  
  > 我们用文件明确状态，人类负责放行，出事能追责。

- **Stage 2**  
  > 即使 Agent 乱来，也改不了不该改的东西。

- **Stage 3**  
  > 系统本身对交付结果负责，而不仅是流程正确。

- **Stage 4**  
  > 这是一个组织级能力，不依赖任何单个人或单个 Agent。

---

### 风险分界线（从哪一 Stage 开始被真正解决）

| 风险 | 从哪一 Stage 开始被真正解决 |
|---|---|
| Agent 自作主张 | Stage 2 |
| 状态被悄悄污染 | Stage 2 |
| 无法解释为什么交付了这个结果 | Stage 1 |
| 交付失败无法回滚 | Stage 3 |
| 团队扩大后流程失控 | Stage 4 |

---

### 当前阶段结论

> **Stage 1 是“真实项目的最小安全形态”。**  
> 再往上升级，带来的不是“更好用”，而是“更贵、更慢、更重”。
